# 蟻鱒鳶ル — ソフトウェア文脈

## コードとしての建築

蟻鱒鳶ルは、設計図を先に完成させてから施工するウォーターフォール型ではない。作りながら考え、作りながら直す。数十年単位で継続するこの工程は、continuous deployment の精神と表面上是重なる。

だが類比には罠がある。アジャイルの「頻繁なリリース」は複数の関与者を前提とする。蟻鱒鳶ルにおいてフィードバックを返すのは誰か。風、雨、苔、錆——非人間的アクターが、むしろ不可避のフィードバックを送り続けている。開放された runtime としての建築という視点が開ける。

## 設計と施工の不可分性

要件定義と実装が同時進行するスクラムの現場と、手練りコンクリートの現場はどこまで重なるか。コンクリートの配合を変えることは、コミットを打つことと同義なのか。違いは明らかだ。git のように `git revert` で巻き戻せない。建築の commit は irreversible だ。

だが、その不可逆性こそが、ソフトウェアが建築の比喩から自己定義してきた60年の歴史を逆手に取るポイントになる。ソフトウェア工学は1968年のNATO会議で「ソフトウェア危機」を建築・土木に倣うべきと論じた。しかし蟻鱒鳶ルは、建築の内部からアジャイル的なプロセスを実践している——これは比喩ではなく、建築自体が比喩を先取りした事例ではないか。

## 技術的負債と未完成

蟻鱒鳶ルには「技術的負債」が存在するのだろうか。むしろ、借金ではない。意図的に完済を遅らせている状態だ。レガシーコードではない。レガシーコードとは「動いているが保守が困難なコード」のことだが、蟻鱒鳶ルは保守そのものが生成活動として機能している。

技術的負債の概念は、完結を前提とするシステムにしか適用できない。完成しない建築に対して「負債」と呼ぶことは、直線的な進歩史観の再導入になる。未完成が戦略であるならば、それは debt ではなく asset だ。

## リファクタリングと改修

リファクタリングは「外部挙動を変えずに内部構造を改善する」作業だが、蟻鱒鳶ルは外部も内部も同時に変化している。防水のパイプコーティングから、多職人によるプルリクエスト的貢献へ——構造の変化は、見た目の変化と不可分だ。

だが、構造的に読み直せば、岡啓輔は数十年かけて「リポジトリのメンテナー」へと変化しているのかもしれない。初期は一人で全てを書いていた。現在は、複数の職人・協力者からの「プルリクエスト」をレビューし、マージするかリジェクトするかを判断している。コードの作者からプロジェクトの保守者へ。これは、建築という不可逆性の高い領域でオープンソース的ガバナンスがどこまで成立するかという問いに直結する。

## マージコンフリクト

複数の職人が同じ建築に介入するとき、建築的「マージコンフリクト」はどう解消されるのか。構造の物理的制約は、Git の conflict marker よりも厳格だ。手練りコンクリートの違いは diff では測れない。触感、温度、湿度、時間——これらはバージョン管理の対象外だ。

ソフトウェアが建築から「工程管理」の比喩を輸入した1960年代と、建築がソフトウェアの「ガバナンス」モデルを参照する現在——この往復運動の果てに、蟻鱒鳶ルはどちらの文脈にも完全に回収されない「宛先不明の手紙」として機能している。
